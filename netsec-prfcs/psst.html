Playground Secure Layer

Abstract

The PLAYGROUND (Poly-Localized Algorithmic Yields Generated Remotely On Unused Network Devices) system is designed to give students and researchers a sandbox in which to develop network security experiments. This standard defines a protocol for secure message delivery called Playground Super Secure Transport, or PSST. PSST is designed to run over PTCL, providing guarantees of mutual authentication and confidentiality on top of the reliability guarantees given by PTCL.

Status of this Memo

This document specifies a Playgrounds standards track protocol for the Playground Experiment, and requests discussion and suggestions for improvements. Distribution of this memo is unlimited.

1.  Introduction

The PLAYGROUND experiment often will have sensitive traffic traveling throughout the network. It is of utmost importance to ensure that parties can speak to one another securely and prevent both passive eavesdropping and active attacks against their communications.

This standard defines a new protocol that runs on top of PTCL called PSST that provides the following guarantees:

  • Confidentiality
  • Authentication
  • Message Integrity

Much like PTCL, PSST, revolves around a statemachine. The session begins with a handshake, encapsulates bi-directional data transfer between the client and server and ends with a teardown routine. Messages within the session are sent with a MAC and encrypted in order to ensure confidentiality and integrity. The handshake involves a mutual authentication process that guarantees authentication to both parties.

This document defines both the state machine of the protocol and the message formats sent between two PSST peers.

3.  Terminology

  • Client: Node initiating connection to another node.
  • Server: Node listening for incoming connection or connections on a C2C port.
  • Peer: The other node in the connection.
  • Sending data: transmitting DATA packets to the peer.
  • Delivering data: returning data (received from the network) to the application layer.
  • Handshake: Initialization routine used to start a PSST session between two peers.
  • Teardown: Ending routine user to end a PSST session between two peers.
  • Session: A single connction between Client and Server. Begins with a Handshake and ends with a Teardown.
  • Session Abort: Termination procedure for when PSST guarantees cannot be met.
  • Cert: A certificate chain that allows one party to authenticate the other. Certificate chains should be laid out [leaf, median cert, median cert, ..., root].

class PSSTHandshakeMessage(MessageDefinition):
  PLAYGROUND_IDENTIFIER = "playground.base.PSSTHandshakeMessage"
  MESSAGE_VERSION = "1.0"

  BODY = [
    ("MessageType", STRING),
    ("Cert_Chain", LIST(STRING)),
    ("Key", STRING),
    ("Nonce_S", STRING),
    ("Nonce_C", STRING),
    ("Data", STRING)]
  ]

Message Field Explanations

  • MessageType: This is a string that denotes the type of message this message is.
  • Cert: This is the certification that the sender of the message is using to authenticate itself to the receiver. In the case of a client to server message, this is the clients certificate and vice-versa.
  • Key: This is the Key that the server will send the client at the end of the Handshake This key MUST be used to encrypt/decrypt all messages exchanged during the data transfer after the Handshake. This MUST be 32 bytes long.
  • Nonce_S: This is a nonce the server uses to challenge the client. Both Nonces MUST be 16 bytes long.
  • Nonce_C: This is a nonce the client uses to challenge the server. Both Nonces MUST be 16 bytes long.
  • Data: This is any other data that is needed to validate the certs, it is implementation dependent based upon what method is being used to validate certificates.

What to fill out when:

  • MessageType is one of "HELLO", "CHALLENGE", "CHALLENGE/RESPONSE", "RESPONSE"
  • If MessageType is "HELLO" then all fields besides "Cert_Chain" SHOULD be zeroed.
  • If MessageType is "CHALLENGE" then "Key" SHOULD be zeroed.
  • If MessageType is "CHALLENGE/RESPONSE" then "Cert_Chain" SHOULD be zeroed.
  • If MessageType is "RESPONSE" then "Cert_Chain" and "Nonce_S" SHOULD be zeroed. 

class PSSTDataMessage(MessageDefinition):
  PLAYGROUND_IDENTIFIER = "playground.base.PSSTDataMessage"
  MESSAGE_VERSION = "1.0"

  BODY = [
    ("MessageType", STRING),
    ("MAC", String),
    ("Data", STRING)]
  ]

Here is the workhorse of PSST that is used to transfer data from one peer to the other. The fields are as follows:

  • MessageType: MUST be set to "DATA", "FIN" or "FINACK". It is used for differentiating between 1) data messages and 2) Teardown.
  • MAC: This is a MAC of the encrypted data. In other words, the data "String" should be encrypted, and a MAC should be taken of the message type concatenated with that encrypted string. If this is not a "DATA" message, then Data SHOULD be zeroed.
  • Data: This is the data being sent from one peer to the other, it should be encrypted using the Key estabished during the Handshake. If this is not a "DATA" message then Data SHOULD be zeroed.

6.  Handshake Specifications

There are 4 messages sent as part of PSST's Handshake, each one with different contents. At the start of the Handshake the client sends the server a (1)HELLO message containing the client's cert. The server responds to this message with a (2)CHALLENGE message with its own cert and a nonce encrypted using the client's cert. The client, upon receiving this message, decrypts the nonce and responds with a (3) CHALLENGE/RESPONSE of that nonce and a new nonce (a challenge for the server) encrypted using the server's cert. The server, upon receiving this message decrypts it and verifies that the first nonce it received is the same nonce as the one it originally sent the client. It responds to the client with a (4) RESPONSE message containing the nonce the client generated and a key to be used in the Steady State communication.

In crypto language this protocol is:

HELLO C -> S: Cert_C

CHALLENGE S -> C: Cert_S,(N_s)_Cert_C

CHALLENGE/RESPONSE C -> S: (N_s, N_c, K_c)_Cert_S

RESPONSE S -> C: (N_c, K_s)_Cert_C

Unlike PTCL, PSST's handshake is an integral part of the guarantees it provides. Within the handshake both parties authenticate themselves to one another using challenge/response mechanisms. The server first challenges the client by encrypting a nonce using the client's cert. The client responds to this challenge and in turn challenges the server, it also sends over the key it will use during data transmission. The server responds to this challenge and sends the client the key that will be used for steady state data transfer after the handshake. If at any point either peer fails the challenge/response the other IMMEDIATELY ends the protocol. Note: if either peer has any reason to suspect the other is fradulent it ends the protocol.

The client and server will also both have to verify that the cert's provided by their peers are valid certs. PSST is not concerned with how this is done, only that it is done.

7.  Data Transmission

During steady state sending, the client and server both use the keys that were exchanged in the handshake to encrypt and decrypt messages. Encryption is done using AES in counter mode, with the nonces acting as IVs - N_s is the server's IV and N_c is the client's IV. IVs should be increased by one after every block is encrypted. The keys will be the keys that were exchanged during the handshake, with the server using K_s to send data to the client and the client using K_c to send data to the server.

After being encrypted the data should be MAC'd using the same key as was used for encryption, this MAC should be attached to the message in the MAC field.

In pseudo code this looks like:

cyphertext = encrypt(plaintext, IV, key)

increment(IV)

data_mac = MAC(cyphertext, key)

message.data = cyphertext

message.mac = data_mac

The receiver will do the reserve of this in order to decrypt the message. It will compare the mac to the cyphertext - if it finds a problem here it will silenty drop the message. Otherwise if the MAC checks out it will decrypt the data of the message and pass it to the layer above it.

8.  Algorithms

For data encryption PSST uses AES256-CTR. The key and IV (nonce) sizes must be what they need to be for AES256, that is, the key must be 32 bytes and the IV must be 16 bytes.

For MAC PSST uses HMAC-SHA256

9.  Teardown

Once one party wants to end the connection it sends a FIN message and waits to receive a FINACK. If a peer receives a FIN it sends back a FINACK and then ends the connection.

This means that whenenever one peer wishes to end a connection and has informed the other peer of its desire to end a connection that connection should end immediately and no more data should be transmitted between the peers.

10.  Assumptions

This protocol makes some fundamental assumptions about the environment it is operating in.

First, it assumes that the peers have some reliable means of communcating, that is, any messages sent by a peer are received in order without errors by the other peer.

Second, it assumes the peers have some method of validating each other's certs. PSST does not specify anything about how this validation is done, but in order for PSST's guarantees to be preserved some method must exist.

This protocol fights against passive attacks because it makes sure that messages no longer travel in clear text. Messages cannot be modified mid-transport without alerting the receiving party which in turn closes the connection. However, this does not mean that an attacker is stopped from using infinite computing power over saved messages to decrypt them.
